# Company Snapshot — Business Reasoning Over HQ

## Why

HQ-037 grounded the Daily Agent on real HQ documents, but the LLM still
received raw, loaded documentation and produced a direct summary of it.
This task is the Runtime's first real **business reasoning** step: the
Runtime itself normalizes, deduplicates, and classifies what is known
and unknown *before* the LLM ever sees anything. The LLM stops
receiving raw documents and starts receiving a structured
**Company Snapshot**.

## Core Types

```
lib/company/company_snapshot.dart
lib/company/company_snapshot_builder.dart
lib/prompts/daily_prompt_builder.dart
```

### `CompanySnapshot`

A normalized, deterministic representation of the company's current
known state. It is not raw documentation and it is not generated by
the LLM. Sections:

- `company`, `products`, `capabilities`, `assets`, `websites`,
  `analytics`, `social`, `services` — deduplicated content per
  category (`capabilities` is derived from the HQ `knowledge/`
  category).
- `knownData` — the HQ categories that had at least one document
  (`Company`, `Products`, `Knowledge`, ...).
- `missingData` — HQ categories with no documents at all, **plus** a
  fixed set of categories the Runtime has no connector for yet at all
  (`Revenue`, `CRM`, `Campaign Performance`, `Search Console`,
  `Finance`, `Customer Metrics`). Unknown data is never fabricated —
  it is always reported here instead.
- `risks` — one deterministic statement per entry in `missingData`.
- `recommendationsInput` — known products and capabilities, prepared as
  raw material for the LLM's recommendations. Not a recommendation
  itself.

### `CompanySnapshotBuilder`

```dart
const builder = CompanySnapshotBuilder();
final snapshot = builder.build(companyContext);
```

Consumes a `CompanyContext` and:

- normalizes and deduplicates documents per category (the same fact
  copied into two files no longer appears twice),
- counts available assets (via each section's length),
- detects which HQ categories are empty,
- detects which categories the Runtime cannot yet report on at all,
- prepares the `recommendationsInput` reasoning material.

Pure and deterministic: no LLM calls, no prompt generation, no
filesystem or network access. The same `CompanyContext` always produces
the same `CompanySnapshot`.

### `DailyPromptBuilder`

```dart
const builder = DailyPromptBuilder();
final prompt = builder.build(
  snapshot: snapshot,
  currentDate: DateTime.now(),
  workflowGoal: 'Generate the daily executive report.',
);
```

Builds the single prompt sent to the LLM from a `CompanySnapshot`, the
current date, and a workflow goal. Prompt generation no longer lives
inside `DailyAgent` — it is its own reusable, independently testable
class. Pure and deterministic: the same three inputs always produce the
exact same prompt string.

## The Daily Agent's Flow

```
Load HQ
   |
   v
Company Context
   |
   v
Company Snapshot
   |
   v
Daily Prompt Builder
   |
   v
LLM
   |
   v
Executive Report
```

`DailyAgent` never touches `CompanyContext` when building the prompt —
only `CompanyContextBuilder` and `CompanySnapshotBuilder` see it. The
prompt is built exclusively from `CompanySnapshot`.

## LLM Responsibility

The LLM is responsible only for **reasoning, prioritization, and
executive communication**. It must never classify documents, count
assets, discover products, or infer missing sections — those
responsibilities belong entirely to the Runtime (`CompanyLoader`,
`CompanyContextBuilder`, `CompanySnapshotBuilder`).

Concretely: the LLM produces the Executive Summary, Company Snapshot
narrative, Products, Business Assets, Current Capabilities, Known
Information, Missing Information, Today's Priorities, and Recommended
Next Actions sections — reasoning only over the `CompanySnapshot` it
was given. The **Data Sources Used** section is never generated by the
LLM: `DailyAgent` appends it itself, directly from
`CompanySnapshot.knownData` / `missingData`, so there is zero risk of
the model inventing which sources contributed.

The first nine sections come from the LLM's response, grounded on the
`CompanySnapshot`. `Data Sources Used` is appended by the Runtime,
deterministically, establishing traceability for future connectors.

## What This Task Does Not Do

- No CRM, Analytics, GitHub, Gmail, Calendar, or Finance connector is
  implemented — `_alwaysMissing` in `CompanySnapshotBuilder` names
  exactly these categories so the report is honest about the gap.
- The Workflow Engine, Tool Registry, Execution Pipeline, and Model
  Providers are untouched.
- No business-specific heuristic lives outside `CompanySnapshotBuilder`
  — `DailyAgent` and `DailyPromptBuilder` only render already-computed
  data; they never decide what is known, missing, or risky themselves.
